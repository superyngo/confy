# i18n (multi-language) plan — TUI + Web/Desktop

**Date:** 2026-07-11 · **Status:** planned, not started
**Initial languages:** `en` (default, canonical) + `zh-TW`. Adding a language later = one more JSON file.

## Goal

All user-facing prose in the TUI and the Web/Desktop UI is translatable and switchable at
runtime. `en` stays the default everywhere so existing tests (which assert English status
strings) keep passing untouched.

## Governing architectural fact

`confy-core`'s `Session` composes user-visible English text **inside core** and ships it to
both UIs through `SessionSnapshot`:

- `session.rs`: ~52 literal `self.status/error = Some("…".into())` sites + ~56
  `format!(…)` sites (grep `self\.(status|error) = Some\(`).
- `node_detail` (session.rs ~line 977): `Path:/Type:/Format:/Value:/Sign:/Comment:` labels.
- `state.rs:70`: `ABOUT_TEXT` const (mirrored by hand in `web/help-content.ts:34`).

Prompt questions are ALREADY enums (`PromptKind` in core state.rs / `PromptView` in view.rs);
each host composes its own question text (`tui/ui.rs:552-567`, `web/prompt.ts:58
promptQuestion`). That pattern is fine — hosts translate those.

**Decision: the translation catalog lives in confy-core** (pure data, `include_str!`, no fs —
satisfies `tests/no_fs_gate.rs`). Session gains a `lang` field; status/error messages route
through a `tr()` lookup. Rejected alternatives: (a) refactor all status into enums — ~100+
sites, large churn, snapshot wire-contract break; (b) heavyweight i18n crates (fluent,
gettext) — wasm-size and dependency cost for two languages is not justified.

## Catalog design

- Repo root `i18n/en.json`, `i18n/zh-TW.json`. Flat string→string maps with namespaced keys:
  `core.*`, `tui.*`, `web.*`. Single source of truth for Rust AND TypeScript (esbuild bundles
  JSON imports natively; Rust embeds via `include_str!` + `serde_json` — already a workspace
  dep of confy-core).
- Parameterized messages use `{0}`/`{1}` positional placeholders with a tiny hand-rolled
  substitute function (no crate). Example:
  `"core.type-change": "type {0} → {1}? y/n"`.
- `en.json` is canonical: `tr` falls back to the `en` entry, then to the raw key, so a missing
  zh-TW entry can never panic or blank the UI.
- Add a small core unit test asserting every key present in `en.json` exists in each other
  language file (or at least that lookups fall back) + placeholder-count parity between
  languages.

**Not translated (by design):** KIND tags (`[T/D]`, `[S:dq  ]`, …— syntax, fixed-width),
key/value content, format names (TOML/JSON/YAML), CLI `--help` (clap; out of scope), and
backend error `Display` strings (`ApplyError` etc. — technical detail interpolated as `{err}`
inside translated wrapper messages like `error: {0}`; translating the wrappers only keeps the
change surgical).

## Phase 1 — core: `Lang` + catalog + status migration

Files: `crates/confy-core/src/session/` (new `i18n.rs`), `session.rs`, `state.rs`,
`intent.rs`, `view.rs`, `dispatch.rs`; new `i18n/*.json`.

1. `i18n.rs`: `pub enum Lang { En, ZhTw }` (serde with rename `"en"`/`"zh-TW"`, `FromStr`,
   `Default = En`); `OnceLock<HashMap<…>>` per language parsed from `include_str!` at first
   use; `pub fn tr(lang, key) -> &str` + `pub fn tr_args(lang, key, &[&str]) -> String`.
2. `Session`: add `pub lang: Lang` + `set_lang`. `Intent::SetLang(String)` (string, not enum,
   keeps the wasm wire contract simple) routed in `dispatch.rs`; `SessionSnapshot` gains
   `lang: String` so hosts stay in sync.
3. Migrate `session.rs` status/error sites to `tr`/`tr_args` under `core.*` keys. Mechanical
   but the bulk of the work; keep message text byte-identical in `en.json` so tests pass.
   Include `node_detail` labels and `format!("{verb} …")`-style composed verbs (give verbs
   their own keys — CJK word order differs, so translate whole sentences, never concatenate
   fragments).
4. `ABOUT_TEXT`: keep as the `en` base; expose `about_text(lang)` (zh-TW body in catalog).
   Note web mirrors it in `help-content.ts` — Phase 3 switches web to the catalog copy.
5. Verify: `cargo test` (all existing English assertions pass with default `En`), new test
   scripting `SetLang("zh-TW")` → a status message asserts the zh-TW string, `cargo clippy
   -- -D warnings`, `cargo fmt`.
6. **Mandatory completion step** (memory: rebuild-wasm-web-after-core-change): rebuild
   `wasm-pack build --target web` in confy-ffi + web bundle **from a scratchpad copy**
   (memory: esbuild deadlocks under /Volumes/Home — copy repo to scratchpad, build, copy
   `web/dist` back) + `tsc` + `node functional_smoke.mjs`.

## Phase 2 — TUI: config file + language picker + About disclosure

Files: `crates/confy-tui/src/` `lib.rs`, `cli.rs`, `tui/app.rs`, `tui/keys.rs`, `tui/ui.rs`,
`tui/mod.rs`; new `config.rs`. New dep: `dirs` (host crate only; core stays fs-free).

1. **Config file** `confy-tui/src/config.rs` (host-owned I/O):
   - Path: `$XDG_CONFIG_HOME/confy/config.toml`, else `~/.config/confy/config.toml`
     (macOS included — terminal-tool convention), Windows `%APPDATA%\confy\config.toml`
     (`dirs::config_dir()` gives APPDATA on Windows; on macOS it gives
     `~/Library/Application Support`, so prefer explicit `$XDG_CONFIG_HOME`/`~/.config` on
     unix and `dirs::config_dir()` on Windows).
   - Content: `lang = "zh-TW"` (room to grow). Parse with the in-repo TOML backend or plain
     string scan — keep it trivial; missing/unparsable file ⇒ defaults, never an error.
   - `load_config() -> Config` + `save_config(&Config)` (create dirs as needed; write is
     best-effort, failure → status message, never a crash).
2. **CLI**: `--lang <code>` on the main command (session-only override, does NOT write the
   config file). Precedence: `--lang` > config file > default `en`.
3. **Picker**: key `L` (verified unbound in `keys.rs`; keymap uses
   `? C E K a c d e f i j k q r s v w x y z`). Small popup listing languages (reuse the
   kind-switch popup pattern: a host-side mini-mode in app.rs is fine — language choice is a
   HOST concern; on select: `session.dispatch(SetLang)` + `save_config` + status confirm).
4. **About**: after the core `about_text(lang)` body, TUI appends host lines:
   `Config: <path>` (the resolved path, shown even if the file doesn't exist yet) and
   `Language: <code>` — path is host-specific so it must NOT go into the core const.
5. **TUI-owned strings** → `tui.*` keys: title bar, column headers, prompt questions
   (`ui.rs:552+`), footer hints, and `keys.rs` help. `help_text(format)` currently returns
   `&'static str` per-format consts — becomes `help_text(format, lang) -> String` (or keeps
   statics per lang; String is simpler). Watch CJK double-width in the fixed-width help
   columns — ratatui handles width, but alignment padding must count display width
   (`unicode-width`), and zh-TW lines must be eyeballed manually.
6. Verify: `cargo test` + clippy; manual TUI run by the user (memory: never drive TUIs via
   pty — hand it over for manual verification).

## Phase 3 — web/touch: catalog + selector

Files: `web/` `host-io.ts`, `ui.ts`, `render.ts`, `panel.ts`, `prompt.ts`,
`help-content.ts`, `convert-dialog.ts`, `typefilter.ts`, `index.html`, `touch/app.ts`,
`touch/render.ts`; new `web/i18n.ts`.

1. `web/i18n.ts`: import both `../i18n/*.json` (esbuild bundles them), `t(key)` /
   `tArgs(key, …)` with the same en→key fallback, `getLang()`/`setLang()` persisted in
   `localStorage["confy-lang"]` (mirror the `confy-theme` pattern in `host-io.ts:188-195`);
   first-run default from `navigator.language` (`zh*` → `zh-TW`, else `en`).
2. Boot + change: after session load, `send({ SetLang: lang })` so core-produced snapshot
   strings match; on selector change, setLang + SetLang intent + full re-render (snapshot
   strings refresh via the dispatch round-trip; static DOM strings re-applied by a
   `applyStaticI18n()` pass).
3. Selector UI: a language button/select next to `btnTheme` in the toolbar (`ui.ts:1167`
   area); touch UI gets the same control in its settings/bottom-sheet surface (respect the
   web-native-ui memory rules: shared modules, all-bottom-sheet on touch).
4. Static strings: `index.html` (~35 labeled elements) via `data-i18n="key"` attributes +
   the `applyStaticI18n()` sweep; TS-composed strings in `ui.ts`/`panel.ts`/`render.ts`/
   `prompt.ts`/`convert-dialog.ts`/`typefilter.ts`/`touch/*` move to `t()` under `web.*`
   keys. `help-content.ts` drops its hand-mirrored About copy and reads the catalog
   (eliminates the existing drift hazard with core's `ABOUT_TEXT`).
5. About/Help on web: state that the language preference lives in browser storage (desktop
   app: the WebView's persistent storage) — no filesystem path to disclose.
6. Verify: `tsc`, rebuild bundle from scratchpad copy, `functional_smoke.mjs`; manual pass
   in browser + `cargo tauri dev` (Tauri shares the same webview localStorage — zero extra
   desktop work).

## Risks / notes

- **Wire contract**: `SessionSnapshot.lang` + `Intent::SetLang` extend the serde contract —
  update `web/types.ts` mirror and `confy-ffi/functional_smoke.mjs`.
- **CJK layout**: TUI help/detail columns and web fixed-width areas need manual zh-TW
  eyeballing; never pad by `str::len`.
- **Sentence composition**: translate whole sentences; audit every `format!` that glues an
  English verb/noun into a template (e.g. `"{verb} {n} node(s) …"` at session.rs:2093 needs
  per-verb full-sentence keys or a `{0} {1} node(s)`-style template that zh-TW can reorder —
  placeholders may appear in any order in a translation).
- **Tests**: default stays `En`; do not rewrite existing assertions. New i18n tests are
  additive.
- **Catalog drift**: the key-parity unit test (Phase 1.item, in core) is the guard.
- Suggested execution: one branch per phase, kept local (memory: keep-branches-local),
  user-triggered merge.
